1. Tips to improve your R workflow

The Main Idea: Reproducibilty

Reproducibility: you, or anyone else, can re-run your analysis from scratch and get the same result.

The tips in the next slides all serve one principle:

Important

Don’t preserve the workspace. Preserve the code that creates the workspace.

Your scripts are the product. Everything else (objects in memory, the console history, the current session) is disposable. See Bryan & Hester, What They Forgot to Teach You About R

Tip 1: Start every session with a clean, empty workspace

When you run an R session, your workspace includes everything created during that session:

  • packages you loaded with library()
  • all user-created objects (data, vectors, functions, models, …)

By default, R offers to save this workspace when you quit and restore it when you return. This is convenient in the short run, but not in the long run.

Tip 1: How?

When closing a session, R asks: “Do you want to save your workspace?” → Answer No.

Even better, change the defaults so R never asks again: Tools → Global Options → General

  • Uncheck “Restore .RData into workspace at startup”
  • Set “Save workspace to .RData on exit” to Never

Tip 1: Why?

A restored workspace can hide objects you forgot about or that your current script never creates:

  • objects created by other scripts may still be lurking
  • packages or data may be loaded that your script never loads
  • you may have modified objects directly in the Console without updating the script

If your script only “works” because of leftovers in the workspace, it is silently broken. That hurts reproducibility

Tip 2: Use a project-based workflow and relative paths

In this class, we use a project-based workflow by creating an .Rproj file in every repo! Value of this approach:

  • keeps all files for one project together in one folder
  • sets the working directory automatically, so relative paths just work
  • lets you switch between projects (or open several) without confusion
Note

If you complete HW5, remember to create .Rproj file in your repo: see these slides from Lecture 10.

Tip 2: Use a project-based workflow and relative paths

Avoid this pattern:

# fragile: works only on this computer, only today
setwd("/Users/sabrina/Documents/css/hw5/data")
df <- read_csv("survey.csv")

An absolute path or a path hard-wired like this example will not work on anyone else’s computer, or on yours a year from now.

Rely on the project root using the .Rproj folders. For more tips, explore the here package builds paths from the top-level project folder, no matter which script or subfolder you are in.

Tip 3: Treat raw data as read-only

Your raw data files are your starting point: never edit them by hand, and never let a script overwrite them.

  • keep raw files in their own folder (e.g., data/raw/) and treat it as untouchable
  • do all cleaning in a script, so every change is recorded and repeatable
  • save cleaned/processed versions to a separate folder (e.g., data/processed/)
Test

If you deleted everything in data/processed/, could you regenerate it by re-running your scripts on the raw data? If yes, your pipeline is reproducible. See Wilson et al. (2017), Good Enough Practices in Scientific Computing

Tip 4: Build debugging skills

What debugging tips have you learned from working with R so far?

Tip 4: Build debugging skills

1. Realize that you have a bug:

  • a bug can be syntactic (like a typo)
  • or conceptual (a misunderstanding of R’s rules)

Tip 4: Build debugging skills

2. Figure out where it is:

  • read your code out loud
  • use print() statements to check each line
  • run one piece of code at a time
  • if it comes from misuse of a specific function, check its documentation with ?function_name
  • ask AI to identify it and explain why
  • take a break!

3. Fix it: then re-run the whole script in a fresh session to confirm

More in “Debugging R code” from What They Forgot to Teach You About R

Tip 5: Read and understand what R tells you

R communicates in three distinct ways, and they are not all bad news:

  • Errors stop your code. R could not do what you asked: this must be fixed.

  • Warnings let the code run, but flag that something may be off. Ceck the result before trusting it.

  • Messages are informative. Nothing is wrong, R is just telling you what it did.

Tip 5: Read and understand what R tells you

log("a")
#> Error in log("a"): non-numeric argument to mathematical function

Errors: code stopped: fix it before moving on

log(-1)
#> Warning message: NaNs produced

Warning: code ran, but the result is NaN (check your input)

library(tidyverse)
#> ── Attaching core tidyverse packages ──────────────

Message: nothing is wrong, just useful info

Tip 6: Restart R often (Session → Restart R)

Stuck with a bug, but you’re 100% sure the code is correct? Restart R:

  • More reliable than rm(list = ls()) or the broom icon. Those only delete user objects; packages you loaded, options you changed, and other session state all survive invisibly
  • restarting keeps RStudio open but clears everything underneath, giving you a truly clean session
  • if your script still works after a restart, run top to bottom, that’s evidence it is reproducible
Note

Restart early, restart often. If restarting feels costly, that is a sign your workflow depends on hidden state, exactly what Tips 1 and 2 prevent.

Tip 7: Use code auto-formatting in RStudio

Consistent style makes bugs easier to see (yours and others). RStudio helps:

  • option 1: Code > Reformat Code (Shift + Cmd/Ctrl + A)
  • option 2: Code > Reindent Lines (Cmd/Ctrl + I)
  • option 3: install the styler package for more on restyling

Open an R script and try this out:

 y<-10
  if (y < 20) {
x <- "Too low" }
else {
  x <-"Too high"}

Recap: habits for reproducible work

  1. Start every session with a clean, empty workspace
  2. Use a project-based workflow (.Rproj in every repo)
  3. Treat raw data as read-only; clean it with scripts
  4. Build debugging skills
  5. Distinguish errors, warnings, and messages
  6. Restart R often
  7. Auto-format your code
Important

FINAL TEST: Close R, reopen the project, run your script top to bottom. Does it work?

2. Deep-dive into R Markdown

R Markdown Recap

R Markdown is a file format (with extension .Rmd) that:

  • combines R code, using code chunks, and text in a single document
  • makes it easy to create reproducible reports, analyses, and presentations
  • you might hear about Quarto (with extension .qmd): a newer alternative to R Markdown (if you learn R Markdown, you can quickly adjust to Quarto)

R Markdown Recap

Three Main Components

  1. YAML header for document title, author, output format, etc. Insert it at the top between ---

  2. Text regular Markdown for headers, lists, links, etc.

  3. Code chunks for code. Insert it with triple backticks ```

Code Chunk Tips

Three ways to insert a code chunk in your R Markdown:

  • Use the RStudio toolbar
  • Type manually ```{r} and ```
  • Use a shortcut
    • Mac: ⌘ + ⌥ + I
    • Windows: Ctrl + Alt + I

R Markdown vs R Script

When to use a script vs. a Rmd

  • When you want to experiment, test functions, debug code

  • When you do not need to communicate output or results to others

  • For scripts that need to run top-to-bottom without narrative

  • For larger projects where you could have some code in R scripts (e.g., data cleaning, analysis pipelines, tests, etc.), and others (e.g., the final-narrated versions) in a Rmd file

R Markdown: What We Know

R Markdown: The Knitting Process

When you “knit” your document, the following sequential things happen, as illustarated here

  1. R Markdown sends the .Rmd file to knitr http://yihui.name/knitr/

  2. Knitr executes all of the code chunks and creates a new plain Markdown .md file which includes the code and its output

  3. This plain Markdown file is then converted by pandoc into any number of output types including html, PDF, Word document, etc.: http://pandoc.org/

R Markdown: Inline Code - How it looks like in your Rmd

---
title: "Gun deaths"
author: "Sabrina Nardin"
output: html_document
---

```{r setup, include = FALSE}
library(tidyverse)
library(rcis)
```

## Gun deaths by age

The `gun_deaths` dataset includes data for `r nrow(gun_deaths)` individuals killed by guns.

```{r youth, echo = FALSE}
youth <- gun_deaths %>% filter(age <= 65)
```

We have `r nrow(youth)` individuals who are 65 or younger. 
Only `r nrow(gun_deaths) - nrow(youth)` are older than 65.

R Markdown: Inline Code - How is rendered once knitted

The gun_deaths dataset includes data for 100798 individuals killed by guns.

We have 85111 individuals who are 65 or younger. Only 15687 are older than 65.

R Markdown: Code Chunks Names

A named code chunk:

```{r youth-distribution}
# code goes here
```

Advantages:

  1. Easier to navigate to specific chunks (using the drop-down code navigator in the script editor). Note: Two code chunks should not have the same name.

  2. Graphics produced by the chunks have useful names

  3. Use it with cache = TRUE to avoid re-performing expensive computations on every run

Tip

cache = TRUE saves a chunk’s output to a file on memory. On future renders, R checks if the code chunk has changed; if not, R skips re-running the chunk and reuses the cached results (see Ch. 27 R for Data Science)

R Markdown: Code Chunk Options

Option What it does Default
eval = FALSE Code is not run, only shown; results do not appear. Useful for code examples or showing code with errors. eval = TRUE
include = FALSE Code is run, but both code and results are hidden. Useful for setup code you don’t want to display. include = TRUE
echo = FALSE Code is run, results are shown, but code is hidden. Good when readers only need the output. echo = TRUE
error = TRUE Code is run and shown, even if it throws an error. Useful when demonstrating broken code. error = FALSE
message = FALSE / warning = FALSE Code is run, but messages and warnings are hidden from the output. message = TRUE / warning = TRUE

Check this table and options for more.

R Markdown: Global Options

Use global chunk options when you want the same settings to apply to ALL code chunks in your R Markdown file.

Global options are defined in a code chunk at the top of your Rmd file with knitr::opts_chunk$set()

For example, if you want to hide the code by default in your entire Rmd:

knitr::opts_chunk$set(
  echo = FALSE    # shows results but hide codes 
)

YAML Header: Date as string

We can add a date as a simple string, and manually modify it every time we modify our file:

---
title: "Gun deaths"
output: html_document
date: "December 1, 2025"
---

But we can also add it automatically!

YAML Header: Date with lubridate

---
title: "Gun deaths"
output: html_document
date: "`r lubridate::today()`"
---

For more on the lubridate package, see Chapter 16 of R for Data Science and Lecture 12 slides

YAML Header: Date with Sys.Date

---
title: "Gun deaths"
output: html_document
date: "`r Sys.Date()`"
---

Sys.Date returns the current day in the current time zone. The default uses the format year/month/day.

YAML Header: Date with Sys.time

---
title: "Gun deaths"
output: html_document
date: "`r format (Sys.time(), '%d, %B, %Y')`"
---

Sys.time returns an absolute date-time value which can be converted to various time zones, and it more easily is customizable. This format says we want the date as: numeric day, full name of the month, and full (4-digit) year, but it can be changed
